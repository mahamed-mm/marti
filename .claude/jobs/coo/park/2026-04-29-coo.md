## Decisions made

- **What**: `.notFound` UX policy: alert before pop, not silent dismiss.
  **Why**: Silent dismiss reads as a phantom nav event. Apple's pattern across Mail/Photos/Maps acknowledges a vanished resource before unwinding. The alert converts a confusing nav event into a deliberate one.
  **Alternatives considered**: Silent dismiss (rejected by design-reviewer); inline error state on detail screen (rejected — listing genuinely gone, staying is worse than unwinding); toast on parent screen post-pop (rejected — wrong location for the acknowledgment).
  **Reversibility**: Cheap. One screen so far.

- **What**: Pushed detail surfaces hide the floating tab bar via `.hideFloatingTabBar(true)`.
  **Why**: Sticky CTAs and immersive content need the screen real-estate that the tab bar consumes. Per-screen opt-in keeps `FloatingTabViewHelper` simple; auto-hide would bake a navigation policy into a layout primitive.
  **Alternatives considered**: Auto-hide on push depth in `FloatingTabViewHelper` (rejected — couples layout to nav state); move policy to `MainTabView` config (rejected — doesn't see pushed children); hide globally with opt-in (rejected — most pushed views in scope today need the bar hidden, but architecture should enforce intent at the screen level).
  **Reversibility**: Cheap. Revisit if >5 detail surfaces accumulate.

Both entries written to `control/decisions.md` 2026-04-28 (date of decision-making, not commit).
