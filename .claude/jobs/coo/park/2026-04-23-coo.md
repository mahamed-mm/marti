## Session summary

One-shot `/fix-bug` session. Report: Discovery screen flashed `EmptyStateView` ("No lists…") on first launch before `SkeletonListingCard` placeholders appeared. Investigation traced the flash to `ListingDiscoveryViewModel.isLoading` defaulting to `false`, making `ListingListView.states` pick the `rails.isEmpty` (empty) branch on the very first render — one SwiftUI tick before `.task` could fire `loadListings()` and flip `isLoading = true`. Applied a one-line fix: default `isLoading = true`. Added a regression test that fails on the old default and passes on the new one. Full `MartiTests` suite green.

## Decisions made

- **What**: Default `ListingDiscoveryViewModel.isLoading` to `true`.
- **Why**: The view already had the correct three-way branch (`isLoading && rails.isEmpty` → skeletons); only the VM's initial state was wrong. One line, no new state machine, no new flags, no existing test broken.
- **Alternatives considered**: `hasLoadedOnce` flag (more state for no benefit); full `LoadState` enum (too big for this bug); set `isLoading = true` as first line inside `loadListings()` (still leaves gap before `.task` fires).
- **Reversibility**: Cheap.

Follow-up, logged but NOT fixed in this session: `DiscoveryView.anchoredItem` has no loading branch — map-mode bottom chrome is blank during first load.
